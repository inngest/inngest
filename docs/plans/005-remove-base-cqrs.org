#+AUTHOR: Darwin D. Wu

#+STATUS: Active

* Summary

We want to remove =pkg/cqrs/base_cqrs= entirely.

The current package mixes unrelated concerns:

- CQRS manager composition
- DB bootstrap and migrations
- history reader/writer implementations
- legacy conversion and normalization leftovers

That boundary no longer matches the architecture introduced by Plan 001.  The
actual backend boundary is now =pkg/db.Adapter= and =pkg/db.Querier=, not
=base_cqrs=.

The target shape is:

#+begin_example
pkg/cqrs/                   interfaces, CQRS-facing types, history payload types,
                            query option structs
pkg/cqrs/manager/           composite cqrs.Manager implementation over db.Adapter
pkg/db/                     backend-neutral persistence contract
pkg/db/sqlite/              SQLite adapter + SQLite bootstrap + SQLite migrations
pkg/db/sqlite/sqlc/         SQLite SQL source and generated query code
pkg/db/postgres/            Postgres adapter + Postgres bootstrap + Postgres migrations
pkg/db/postgres/sqlc/       Postgres SQL source and generated query code
#+end_example

* Decisions

** Decision 1: remove =pkg/cqrs/base_cqrs= entirely

We are not keeping a deprecated forwarding layer.  The package should disappear
once the replacement packages exist and imports are updated.

Reasoning:

- the current package name no longer describes its responsibility
- retaining forwarding wrappers would prolong the wrong abstraction boundary
- import churn is manageable because external entrypoints are limited

** Decision 2: CQRS manager belongs under =pkg/cqrs=

The composite manager implementation should live in =pkg/cqrs/manager=.

Reasoning:

- =pkg/cqrs= already defines the public contracts
- the manager implements CQRS behavior, not backend bootstrap
- callers should think in terms of "CQRS manager over an adapter", not
  "base cqrs"

** Decision 3: CQRS owns history reads, writes, and payload types

History is part of the data access layer.  Callers should not need to depend on
separate reader or writer contract packages in order to access persistence.

Reasoning:

- CQRS is the caller-facing DAL boundary
- history reads are just another DAL read surface
- history writes are still persistence operations
- callers should depend on =pkg/cqrs= contracts rather than implementation-side
  or adjacent packages
- CQRS should own the payload and result types used by those contracts

Implications:

- move the history read interfaces into =pkg/cqrs=
- move the history write interfaces into =pkg/cqrs=
- move payload and result types currently owned by =pkg/history_reader= and
  =pkg/execution/history= into =pkg/cqrs=
- make =pkg/cqrs/manager= implement the history read and write surfaces
- deprecate =pkg/history_drivers= entirely
- replace the devserver's parallel in-memory history store with CQRS-backed
  history as the single source of truth
- define narrower history interfaces in =pkg/cqrs= and embed them in
  =cqrs.Manager= rather than flattening all history methods directly into the
  top-level manager declaration
- keep only minimal temporary compatibility shims for =pkg/history_reader= and
  =pkg/execution/history= during migration
- mark every temporary shim as =DEPRECATED= and track it for fast-follow removal
- have executor lifecycle code depend directly on the narrower CQRS history
  interfaces in the end state

** Decision 4: bootstrap and migrations belong in each DB package

Bootstrap and migration ownership should move to =pkg/db/sqlite= and
=pkg/db/postgres= instead of a CQRS package.

Reasoning:

- opening a SQLite connection is a SQLite concern
- opening a Postgres connection is a Postgres concern
- embedding and running migrations is dialect-specific
- CQRS should accept an adapter and remain agnostic to connection lifecycle

This separates two concerns:

- dialect-specific bootstrap
- app-level backend selection

Dialect-specific bootstrap belongs in =pkg/db/{dialect}=.  App-level selection
can remain in callers such as =pkg/devserver= or, if needed later, a thin
factory package outside =pkg/cqrs=.

Decision for first pass:

- keep backend selection in callers
- do not introduce =pkg/db/open= initially
- revisit only if caller duplication becomes a concrete maintenance problem

** Decision 5: initial refactor is structural, not behavioral

The first pass should preserve behavior.  We should not combine package moves
with query rewrites, API changes, or storage model changes unless required to
complete the move.

** Decision 6: sqlc packages move with their dialect adapters

Generated query packages and their SQL source should move out of
=pkg/cqrs/base_cqrs/sqlc= and live with the dialect packages that own them.

Reasoning:

- the generated code is backend-specific, not CQRS contract code
- colocating sqlc code with adapters makes dialect ownership explicit
- removing =base_cqrs= should also remove its generated-code subtree
- keeping sqlc under =base_cqrs= would preserve the wrong package boundary

** Decision 7: data access changes must be backed by strong tests

This refactor touches the persistence foundation of the system.  Every
structural move should preserve or improve test coverage for the DAL so that
callers can rely on this layer without re-litigating its correctness.

Reasoning:

- =pkg/cqrs= and =pkg/db= are foundational packages for higher layers
- broad import churn increases regression risk even when behavior is intended to
  stay the same
- dialect parity, transaction semantics, and history correctness need explicit
  coverage, not implicit confidence
- the goal is for the DAL to be treated as reliable infrastructure, not a layer
  that callers work around or distrust

Testing requirements:

- preserve existing DAL coverage while moving code
- add targeted tests when a move changes constructor shape, ownership, or
  package boundaries
- run focused DAL and history regression tests after each migration phase
- keep SQLite and Postgres behavior aligned through cross-dialect checks where
  applicable
- treat missing or weakened data-access coverage as a blocker for completing a
  phase

* Current State

Today =pkg/cqrs/base_cqrs= contains:

- =base_cqrs.go=: DB opening + migration execution
- =cqrs.go=: a ~3.3k-line composite manager implementing multiple CQRS domains
- =history_driver.go=: =execution/history.Driver= implementation
- =history_reader.go=: =history_reader.Reader= implementation
- =sqlite_converter.go=: CQRS/domain conversion helpers with stale naming
- =sqlc/postgres/db_normalization.go= and =sqlc/postgres/normalization.go=:
  leftover SQLite-first normalization paths that no longer fit the intended
  architecture

The main public entrypoints in use are:

- =base_cqrs.New()=
- =base_cqrs.NewCQRS()=
- =base_cqrs.NewHistoryDriver()=
- =base_cqrs.NewHistoryReader()=

This means the migration surface is broad in test imports but narrow in actual
API shape.

Related history state today is also split across multiple packages:

- =pkg/history_drivers/memory_writer= stores execution history in an in-memory
  singleton map
- =pkg/history_drivers/memory_reader= reads from that same in-memory map
- devserver currently wires both the CQRS-backed history driver and the memory
  writer into the execution lifecycle listener
- devserver injects the memory reader into =coreapi= for parts of the GraphQL
  surface

This means devserver currently maintains two history backends:

- the normal CQRS-backed store
- a dev-only in-memory mirror

The target architecture should remove that duplication.

* Target Package Responsibilities

** =pkg/cqrs=

Owns:

- public interfaces
- CQRS-facing structs returned to callers
- history read interfaces
- history write interfaces
- history payload and result types
- shared query and filter option types

Does not own:

- connection opening
- migrations
- dialect-specific sqlc usage

** =pkg/cqrs/manager=

Owns:

- the composite implementation of =cqrs.Manager= and =cqrs.TxManager=
- transaction scoping over =db.Adapter=
- CQRS-side mapping from =pkg/db= domain rows into =pkg/cqrs= result types
- domain-specific manager logic currently concentrated in =base_cqrs/cqrs.go=

Suggested internal file split:

- =manager.go=
- =apps_functions.go=
- =events.go=
- =runs_history.go=
- =queue_snapshots.go=
- =traces.go=
- =connect.go=
- =convert.go=
- =history.go=

** =pkg/db/sqlite= and =pkg/db/postgres=

Each dialect package owns:

- adapter construction
- connection opening helpers
- migration execution
- embedded migration files
- sql source and generated query code
- dialect-specific sqlc adaptation into =pkg/db= domain rows and params

Possible API shape:

- =Open(ctx, opts) (*sql.DB, error)=
- =Migrate(ctx, conn) error=
- =New(conn) *Adapter=

The exact API names can be decided during implementation, but ownership should
remain dialect-local.

* Risks

** Transaction semantics regress

Risk:

- =WithTx()=, =Commit()=, and =Rollback()= behavior may drift during package
  moves

Mitigation:

- move transaction code intact first
- keep existing CQRS transaction tests green before deeper file splitting

** Trace and span behavior regresses

Risk:

- trace/span logic is the most specialized and interdependent portion of the
  manager

Mitigation:

- move traces last
- preserve helper functions and tests before attempting local cleanup

** History ownership becomes unclear

Risk:

- history read/write code could be added to =manager= in an ad hoc way and
  recreate the same package bloat under a new name

Mitigation:

- keep history contracts and payload types in =pkg/cqrs=
- keep history implementation isolated in dedicated manager files
- avoid mixing history logic across unrelated domain files

** Devserver history migration regresses UI behavior

Risk:

- devserver currently relies on a separate memory-backed history path for parts
  of GraphQL and cancel-run behavior

Mitigation:

- switch devserver API reads to CQRS-backed history before removing
  =pkg/history_drivers=
- verify run-details, cancel-run, and rerun flows against the dev-server UI
- remove dual-write only after CQRS-backed reads are confirmed equivalent

** Dialect bootstrap APIs diverge too much

Risk:

- independent SQLite/Postgres bootstrap helpers may drift in ergonomics

Mitigation:

- align option and function naming where it makes sense
- allow small differences where dialect needs actually differ

** Data access regressions slip through structural churn

Risk:

- package moves, constructor changes, and import rewrites can quietly weaken
  DAL guarantees even when the refactor is intended to be behavior-preserving

Mitigation:

- define a DAL verification suite up front and keep it green throughout
